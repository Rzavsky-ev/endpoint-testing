# API Automation Tests

## Описание проекта

Проект предназначен для автоматизированного тестирования веб-сервиса, который взаимодействует с
внешними сервисами аутентификации и выполнения действий. Тесты покрывают три основных функциональных блока:
вход в систему (LOGIN), выполнение действий (ACTION) и выход из системы (LOGOUT).

## Структура проекта

```text
src/test/java/
├── api/
│ ├── base/ # Базовые классы для тестов
│ ├── tests/ # Тестовые классы
│ ├── utils/ # Вспомогательные утилиты
│ ├── specs/ # Спецификации запросов/ответов
│ └── exceptions/ # Пользовательские исключения
```

## Используемые технологии

- Java 17
- JUnit 5
- Rest Assured
- WireMock
- Allure
- Faker
- Maven

## Запуск тестов

```bash
# Все тесты
mvn clean test

# Запуск с генерацией Allure отчета
mvn clean test allure:serve

# Запуск конкретного тестового класса
mvn -Dtest=LoginTest test 

# Запуск конкретного теста
mvn -Dtest=LoginTest#successfulLoginWithValidToken test
```

## Allure отчеты

После запуска тестов с `allure:serve` отчет автоматически откроется в браузере.

## Компоненты проекта

### BaseTest (api.base)

Базовый класс для всех тестов. Управляет жизненным циклом WireMock сервера:

- Запуск сервера перед всеми тестами

- Сброс заглушек перед каждым тестом

- Остановка сервера после всех тестов

### Тестовые классы

#### LoginTest — тесты для входа в систему

✓ Успешный вход с валидным токеном

❌ Вход с невалидными токенами (параметризованный тест)

❌ Обработка ошибок внешнего сервиса

❌ Вход без API ключа и с неверным ключом

❌ Повторный вход с тем же токеном

❌ Вход без параметра action

#### ActionTest — тесты для выполнения действий

✓ Успешное действие после входа

❌ Действие с невалидным токеном

❌ Действие без предварительного входа

❌ Действие после выхода

❌ Обработка ошибок внешнего сервиса

✓ Несколько действий подряд

❌ Действие без API ключа и с неверным ключом

❌ Действие без параметра action

#### LogoutTest — тесты для выхода из системы

✓ Успешный выход после входа

❌ Выход с невалидным токеном

❌ Выход без предварительного входа

❌ Повторный выход

✓ Полный жизненный цикл токена

✓ Многопользовательский сценарий

❌ Выход без API ключа и с неверным ключом

### Утилитные классы

#### Constants (api.utils)

- Централизованное хранение всех констант:

- URL и порты

- Эндпоинты

- Заголовки и параметры

- HTTP статусы

- Сообщения об ошибках

- Тестовые данные

#### TokenGenerator (api.utils)

Генерация тестовых токенов:

- `generateValidToken()` — валидный токен (32 символа, A-F0-9)
- `generateTokenOfRandomLength()` — токен случайной длины
- `generateLowerCaseToken()` — токен со строчной буквой
- `generateTokenWithSpecialChars()` — токен со спецсимволами
- `invalidTokenDataProvider()` — DataProvider для параметризованных тестов

#### WireMockStubBuilder (api.utils)

Настройка заглушек WireMock:

- `mockAuthSuccess()` — успешный ответ от /auth
- `mockAuthError()` — ошибочный ответ от /auth
- `mockDoActionSuccess()` — успешный ответ от /doAction
- `mockDoActionError()` — ошибочный ответ от /doAction

#### AllureReporter (api.utils)

Работа с Allure отчетами:

- `addTestData()` — добавление вложений
- `getStatusText()` — преобразование HTTP кодов в текст

#### ActionSteps (api.utils)

Выполнение основных действий:

- `performLogin()`, `performAction()`, `performLogout()` — выполнение запросов
- `performSuccessfulLogin()`, `performSuccessfulAction()`, `performSuccessfulLogout()` — успешные сценарии с
  проверками

#### ResponseVerifier (api.utils)

Проверка ответов API:

- `verifyLoginSuccess()` — проверка успешного входа
- `verifyActionSuccess()` — проверка успешного действия
- `verifyLogoutSuccess()` — проверка успешного выхода
- `verifyInvalidTokenError()` — проверка ошибки формата токена
- `verifyTokenNotFoundError()` — проверка ошибки "токен не найден"
- `verifyInvalidApiKeyError()` — проверка ошибки API ключа
- `verifyMissingActionError()` — проверка отсутствия параметра action
- `verifyTokenAlreadyExistsError()` — проверка повторного использования токена
- `verifyExternalServiceError()` — проверка обработки сбоев внешнего сервиса

### Спецификации

#### RequestSpec (api.specs)

Построение спецификаций запросов:

- `getBaseSpec()` — базовая спецификация
- `forEndpoint()` — спецификация для конкретного эндпоинта
- `forValidApiKey()`, `forWrongApiKey()`, `forEmptyApiKey()` — спецификации с разными API ключами

#### ResponseSpec (api.specs)

Построение спецификаций ответов:

- `forSuccess()` — успешный ответ
- `forError()` — ошибочный ответ
- `forValidationError()` — ошибка валидации
- `forTokenNotFoundError()` — ошибка "токен не найден"
- `forInvalidApiKeyError()` — ошибка API ключа

### Тестовые сценарии

#### Успешные сценарии

✅ Вход в систему с валидным токеном

✅ Выполнение действия после входа

✅ Выход из системы после входа

#### Негативные сценарии

❌ Все виды невалидных токенов (короткие, со строчными буквами, со спецсимволами, пустые, null)

❌ Отсутствие API ключа

❌ Неверный API ключ

❌ Отсутствие параметра action

❌ Повторный вход с тем же токеном

❌ Действие без входа

❌ Действие после выхода

❌ Повторный выход

❌ Выход без входа

#### Сценарии с внешними сервисами

⚠️ Ошибки внешнего сервиса при входе (403, 404, 500)

⚠️ Ошибки внешнего сервиса при выполнении действия (403, 404, 500)

#### Комплексные сценарии

🔄 Полный жизненный цикл токена: LOGIN → ACTION → LOGOUT → проверка неработоспособности

👥 Многопользовательский сценарий с двумя токенами

### Теги

- `SMOKE` — критические сценарии, проверяющие основную функциональность
- `REGRESSION` — полный набор регрессионных тестов
- `NEEDS_CLARIFICATION` — тесты, требующие уточнения у разработчиков

## ⚠️ Известные проблемы и расхождения

### Требования к формату токена

**В задании указано:** токен должен состоять из символов `A-Z0-9` (буквы A-Z и цифры)

**Фактическая реализация:** приложение принимает только символы `A-F0-9` (только буквы A-F и цифры)
